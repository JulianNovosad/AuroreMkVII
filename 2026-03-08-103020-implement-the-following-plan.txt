  So the picture is: MkVI has a working fire control loop (even if detection is

  ---
  Three Approaches to Making MkVII a True Upgrade

  Approach A: "Port and Polish" вҖ” Implement all MkVI features, RT-correct

  Take every feature from MkVI's logic.cpp (2338 lines), port it into MkVII's
  modular architecture, test it properly, and wire everything up. Preserve
  MkVI's feature parity: RK4 ballistics, IoU tracker, gimbal controller,
  display, config loader, everything.

  Pro: Complete feature parity + better RT architecture = definitively better
  system. Tests can be 1:1 mapped.
  Con: ~6-8 weeks of implementation. Some MkVI "features" are dead code or stubs
   вҖ” you'd be porting them anyway.
  Risk: Low. Well-understood scope.

  Approach B: "Selective Upgrade" вҖ” Implement only what's needed to run 
  end-to-end, add one genuine upgrade

  Wire the minimum viable fire-control pipeline (gimbal controller, state
  machine integration, config loader, basic HUD), then add one thing MkVI never 
  had: either color segmentation as a second detection path (parallel to ORB,
  for colored targets), or a proper Kalman filter replacing the EMA tracker.

  Pro: Gets MkVII running end-to-end fast. One genuine upgrade over MkVI is a
  clear selling point.
  Con: Still missing uncertainty propagation, sensor fusion, ZMQ telemetry. Test
   coverage gap remains.
  Risk: Low-medium. Scope is bounded.

  Approach C: "1:1 Test Coverage First, Then Build" вҖ” Test-driven implementation

  Map every MkVI test to an equivalent MkVII test first (coverage spec: 44 tests
   вҶ’ 44+ MkVII CTest equivalents), then implement to make them pass. This forces
   the question of what MkVII actually needs vs. what MkVI had as dead weight.

  Pro: Forces disciplined scope. Every line of implementation code serves a
  test. No orphaned stubs.
  Con: Requires clear translation of MkVI tests to MkVII's modular structure
  first.
  Risk: Low. This is how the "excellent bones" should be finished.

  ---
  My recommendation: Approach C first, then B.

  Write the test specs (mapping MkVI's 22 real tests to MkVII equivalents), then
   implement iteratively until all pass. The one genuine upgrade to add is a
  local HUD (since MkVI's display requires fbdev/DRM hardware, and MkVII already
   has OpenCV вҖ” a cv::imshow debug window is trivially portable and works on
  laptop). This is a better laptop dev UX story than MkVI ever had.
